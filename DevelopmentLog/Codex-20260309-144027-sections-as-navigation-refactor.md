# 栏目即导航收口记录

## 背景

项目之前同时维护了两套接近重复的入口模型：

1. `sections`
   - 表示栏目
   - 被分类、文章编辑页引用
2. `nav`
   - 表示前台导航
   - 其中大部分条目其实和栏目一一对应

这会导致后台要同时维护“栏目页”和“导航页”两份重复数据，只是导航里额外多了“首页”和“关于我”两个系统入口。

## 本次调整

这次把模型收口成：

- 栏目 `sections` 是唯一真实来源
- 前台导航由以下规则自动生成：
  - 首页
  - 所有已启用栏目
  - 关于我

也就是说，导航不再是单独手工维护的一套结构，而是栏目模型的派生结果。

## 具体改动

### 1. 服务端导航改为派生数据

修改文件：

- `scripts/admin-server.cjs`

改动内容：

- 新增 `buildNavigationItems()`，统一从 `sections` 生成导航数据
- 固定插入两个系统链接：
  - `nav-home`
  - `nav-about`
- `GET /api/nav` 现在返回派生后的导航，而不是直接读取 `nav.json`
- `GET /api/data/nav` 现在也返回派生导航
- `PUT /POST /PATCH /DELETE /api/data/nav` 已禁止，避免继续手工维护重复导航
- `dashboard` 的导航统计也改为基于派生导航计算

### 2. 后台设置页取消手工导航编辑

修改文件：

- `admin/settings.html`

改动内容：

- 移除了“新增导航项 / 删除导航项 / 手工编辑 href / 排序 / 可见性”这套表单
- 设置页中的导航模块改为“自动生成导航预览”
- 文案明确说明：
  - 首页和关于我是系统链接
  - 启用中的栏目会自动出现在中间
  - 栏目本身要去栏目页维护
- 设置保存时只保存 `settings`，不再向 `/api/data/nav` 提交内容

### 3. 前台默认导航回退逻辑更新

修改文件：

- `src/constants/blog.js`

改动内容：

- 默认导航从旧的“分类链接”回退，改成与当前模型一致的：
  - 首页
  - 设计创作
  - 技术笔记
  - 工具分享
  - 问题记录
  - 生活随笔
  - 关于我

这样即使接口异常，前台回退行为也和当前栏目路由保持一致。

### 4. 初始化与冒烟测试同步更新

修改文件：

- `scripts/init-data.cjs`
- `scripts/smoke-test.cjs`

改动内容：

- `init-data` 中的 `nav` 默认数据改为空数组，并说明导航由栏目派生
- `smoke-test` 不再测试手工写入导航
- `smoke-test` 改为校验：
  - 生成导航数量 = 启用栏目数 + 2
  - 生成导航中必须包含首页和关于我
  - dashboard 的导航统计与派生导航一致

## 当前结果

经过这次调整后：

- 后台不需要再维护重复导航
- 栏目页成为唯一真实入口源
- 首页和关于我被明确为系统链接
- 前台和后台的导航来源统一了

## 验证

本次已执行并通过：

- `node --check scripts/admin-server.cjs`
- `npm run lint`
- `npm run validate:data`
- `npm run build`
- `npm run smoke:test`

## 后续建议

这次是把“配置模型”先收口了，但前台内部仍有一部分地方在用栏目名称去辅助推断内容范围。

下一步更稳的方向是：

1. 前台栏目态改为显式依赖 `section.id / section.slug`
2. 分类和标签的“父级归属”从字符串名称逐步迁移到稳定 ID
3. 最终彻底把 `nav.json` 退役为历史兼容文件
