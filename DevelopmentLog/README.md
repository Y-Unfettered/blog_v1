# my_blog 项目总结与后续开发计划

## 1. 项目现状总结

`my_blog` 当前已经是一个可以运行的个人博客 MVP，包含以下几块：

- 前台博客站点：基于 Vue 3 + Vite
- 后台管理：基于原生 HTML + JS 页面
- 简易 API：基于 Node.js 原生 `http`
- 数据存储：本地 JSON 文件
- 部署方式：Docker + Nginx

从现状看，这个项目已经完成了“能展示、能管理、能部署”的基础闭环，但还没有进入稳定的工程化阶段，整体更接近“功能已堆起来、架构待收口”的状态。

## 2. 当前优点

- 主站功能已经比较完整，包含首页、文章详情、栏目页、标签页、关于页等内容形态。
- 数据结构已经覆盖文章、分类、标签、导航、栏目、设置等核心对象。
- 后台已经具备文章、分类、标签、栏目、设置等管理页面雏形。
- Docker 部署链路已经具备基础框架，适合继续往线上环境推进。

## 3. 当前主要问题

### 3.1 前台架构过重

当前主站大量状态和页面逻辑集中在 `src/App.vue`，文件体量已经很大。继续在这个文件上叠功能，会导致：

- 维护成本上升
- 回归风险变大
- 功能之间的状态耦合越来越重

### 3.2 前后台接口契约不统一

后台部分页面使用的是 `/api/categories`、`/api/settings` 这样的接口路径，但服务端真正支持完整增删改查的是 `/api/data/*` 体系。部分页面读取和写入的数据结构也不一致，这会导致：

- 某些后台页面看起来存在，但实际不稳定
- 后续开发需要不断打补丁
- 调试成本增加

### 3.3 数据模型存在历史兼容痕迹

当前文章数据里同时存在多组字段，例如：

- `categories` 和 `categoryIds`
- `tags` 和 `tagIds`
- `created_at` 和 `createdAt`

这说明项目已经经历过多轮结构调整，但还没有彻底完成数据模型统一。

### 3.4 部署和安全配置不够严谨

API 服务存在默认管理员账号、默认密码和默认 JWT 回退逻辑。对于本地开发问题不大，但如果直接进入生产，会有明显风险。

### 3.5 工程链路分裂

项目既使用了 Vite 构建，又在主页和后台大量使用 CDN 方式引入 Tailwind 和其他资源。长期来看会带来：

- 样式来源分散
- 构建结果不可控
- 本地与生产环境表现不一致

### 3.6 仓库中保留了较多历史试验文件

后台目录里有多份 `.bak`、`index-v2.html`、`index-new.html`、`index-single.html` 等历史文件，会影响判断当前真实入口，也会增加后续维护噪音。

## 4. 开发优先级建议

建议优先顺序如下：

1. 修正确性问题
2. 收口后台接口与数据结构
3. 拆分前台架构
4. 补后台真实数据能力
5. 补工程化与上线能力

## 5. 分阶段后续开发计划

### 第一阶段：修基础正确性

目标：先让“现在已有的功能”真正稳定可用。

建议任务：

- 统一后台接口调用，全部对齐到一套明确的 API 规则
- 修复 `categories`、`settings` 等页面的数据读写不一致问题
- 处理独立文章页遗留逻辑，明确是否继续保留 `article.html`
- 补齐 `sections` 初始化逻辑
- 清理明显无用的历史试验文件和备份文件
- 收紧默认管理员和 JWT 的环境变量策略

预估：2 到 4 天

### 第二阶段：做前台架构整理

目标：让主站从“单文件大应用”变成可持续开发的结构。

建议任务：

- 将 `src/App.vue` 拆成 `views`、`composables`、`services/api`
- 引入 Vue Router，替代目前的 hash 路由控制
- 统一文章、分类、标签、设置的数据模型
- 补前台请求层封装和错误处理
- 做代码分割，降低首屏 JS 包体积

预估：4 到 7 天

### 第三阶段：补后台可用性

目标：让后台从“页面集合”变成“真正可运营的管理系统”。

建议任务：

- 将后台仪表盘改为真实数据驱动
- 提取后台公共布局、导航和请求封装
- 完善文章编辑器的数据保存、预览和校验
- 让设置页真正控制前台展示内容
- 增加栏目、标签、分类之间的关联约束

预估：4 到 6 天

### 第四阶段：补工程化与上线能力

目标：为后续长期维护和部署做准备。

建议任务：

- 增加 lint、基本测试和构建校验
- 增加 JSON schema 或数据校验逻辑
- 增加数据备份与恢复方案
- 清理 CDN 依赖，逐步统一到正式构建链
- 如果文章规模继续增长，考虑迁移到 SQLite

预估：5 到 7 天

## 6. 推荐近期执行顺序

如果按“投入最少、收益最大”的角度，建议直接按下面顺序推进：

1. 修后台接口不一致
2. 修部署安全配置
3. 拆前台 `App.vue`
4. 清理历史遗留入口和试验页面
5. 把后台仪表盘改成真实数据
6. 再补测试和工程化
